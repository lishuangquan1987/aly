## 🔴 严重问题 (Critical)

### C1. deploy-server.py 硬编码服务器 IP、用户名和真实姓名

- **文件**: `deploy-server.py`，第 4-8 行
- **严重程度**: 🔴 安全 — 凭据泄露

```python
HOST = "10.96.115.14"
USER = "quartecs"
REMOTE_DIR = "/home/quartecs/lishuangquan/aly"
PORT = 7000
```

内网 IP、Linux 用户名、以及真实姓名（`lishuangquan`）已提交到 Git 历史中。应改用环境变量或 `.env` 文件（加入 `.gitignore`）。

---

### C2. deploy-server.py 密码通过命令行参数传递

- **文件**: `deploy-server.py`，第 30 行
- **严重程度**: 🔴 安全 — 凭据泄露

```python
password = sys.argv[1]
```

命令行参数在大多数操作系统上可通过 `ps`、`/proc/*/cmdline` 等被其他用户看到。应仅使用 `getpass`（第 32 行的 fallback 方式更好）。

---

### C3. deploy-server.py SSH 自动接受任何主机密钥（MITM 风险）

- **文件**: `deploy-server.py`，第 35 行
- **严重程度**: 🔴 安全

```python
ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
```

`AutoAddPolicy` 盲目信任任何主机密钥，存在中间人攻击风险。至少应使用 `WarningPolicy`，生产环境应使用 `RejectPolicy` + known_hosts 文件。

---

### C4. deploy-server.py 使用密码认证而非密钥认证

- **文件**: `deploy-server.py`，第 36 行
- **严重程度**: 🔴 安全

```python
ssh.connect(HOST, username=USER, password=password, timeout=30)
```

自动化部署应使用 SSH 密钥认证，而非密码认证。

---

### C5. deploy-server.py 使用 kill -9 无优雅关闭

- **文件**: `deploy-server.py`，第 42 行
- **严重程度**: 🔴 运维

```python
ssh.exec_command(f"lsof -ti :{PORT} | xargs -r kill -9 2>/dev/null; sleep 2; echo OK")
```

直接发送 `SIGKILL`，不先尝试 `SIGTERM`。服务器无法执行任何清理（关闭 DB 连接、刷新缓冲区、写状态）。

### C6. Server: AfterApplyUpdateScript 无验证 — 远程代码执行风险

- **文件**: `server/controllers/project_controller.go`，第 141 行；`server/internal/service/project_service.go`，第 163 行
- **严重程度**: 🔴 安全 — RCE

`AfterApplyUpdateScript` 字段（schema 注释为"更新后执行的脚本"）在存储时无任何路径清理、扩展名检查或内容验证。如果该脚本后续在客户端被执行，这是严重的远程代码执行风险。

### C7. Client: AfterApplyUpdateScript 路径穿越攻击

### C8. Client: rollback.go 静默忽略 os.Rename 失败

## 🟠 高优先级 (High)

### H1. Server: 所有 HTTP 错误返回 200 状态码

- **文件**: `server/controllers/project_controller.go`（普遍存在）、`file_download_controller.go`、`file_upload_controller.go`
- **严重程度**: 🟠 架构 — HTTP 语义丢失

所有错误（验证失败、数据库错误、文件未找到）都返回 `ctx.JSON(200, ...)` 配合 `isSuccess: false`。客户端无法通过 HTTP 状态码区分成功和失败，依赖 HTTP 4xx/5xx 的中间件/代理/监控全部失效。

### H2. Server: 全局可变状态 db.Client + 无依赖注入

- **文件**: `server/internal/db/db.go`，第 19 行；`server/internal/service/project_service.go`（普遍存在）

var Client *ent.Client  // 全局可变状态

所有 service 和 controller 层直接引用全局 `db.Client` 和 `db.WithTx`。没有任何接口定义，无法进行单元测试或 mock。整个代码库零接口。

### H4. publish-gui: 命令行注入风险（多处）

- **文件**: `publish/publish-gui/src/AlyPublish/Services/CliService.cs`，第 108, 118, 122, 140-143 行
- **严重程度**: 🟠 安全 — 命令注入

```csharp
// 只转义了双引号，cmd.exe 还会解释 %, ^, &, |, <, > 等
$"add {string.Join(" ", files.Select(f => $"\"{f.Replace("\"", "\\\"")}\""))}"
```

文件名、版本号、消息等用户输入在拼接到 CLI 参数时，仅转义了双引号。Windows cmd.exe 中 `%`、`^`、`&`、`|`、`>`、`<` 等字符都可能导致命令注入。

### H5. publish-gui: ConfigService 无线程安全 + 凭 DisplayName 匹配项目

- **文件**: `publish/publish-gui/src/AlyPublish/Services/ConfigService.cs`
- **严重程度**: 🟠 数据丢失 + Bug

问题 1 — 无锁：所有方法（Load、Save、Add、Remove、Update）操作同一文件无锁保护。两个 Tab 同时保存会互相覆盖。

问题 2 — `RemoveProject` 用 `RemoveAll(p => p.DisplayName == displayName)`，如果两个项目同名（用户可配置），两个都会被删除。

问题 3 — `UpdateProject` 凭 `DisplayName` 查找，如果 DisplayName 被修改了，`FindIndex` 失败，更新被静默跳过。

---

### H6. publish-gui: GUI 将 CLI 作为子进程包装而非库链接

- **文件**: `publish/publish-gui/` 整体架构
- **严重程度**: 🟠 架构

每个操作（status、add、push、config）都 spawn 一个新的 `Process`。带来的问题：
- 慢（每次操作都有进程启动开销）
- 脆弱（依赖 CLI 输出格式精确匹配；CLI 版本不匹配会导致 GUI 中断）
- 难以调试（两个进程边界）
- 无共享状态
