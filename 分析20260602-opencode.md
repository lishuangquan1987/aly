## 一、严重 (Critical) — 需立即修复

### 1. client: 路径分隔符不匹配导致所有子目录文件误判为"已变更"

- **文件**: `client/util/file.go:124`, `client/cmd/check_diff.go:74`, `client/cmd/download_update.go:101`
- **问题**: `LocalFileMD5Map()` 用 `filepath.Rel()` 生成的 key 是 `subdir\file.txt`（Windows 反斜杠），但 `normalizePath()` 将服务端路径转为 `subdir/file.txt`（正斜杠）。map 查找永远失败，导致所有子目录文件每次都被当作"不同"或"缺失"。
- **影响**: `check_diff` 报假阳性；`download_update` 重复下载已有文件，严重浪费带宽。
- **修复**: 在 `LocalFileMD5Map()` 中将 `relPath` 统一转为正斜杠，或在比对前对两侧都做 `normalizePath`。

### 2. client: `WriteVersion` 缺少 `fsync`，断电可致 version.json 损坏

- **文件**: `client/config/version.go:67-90`
- **问题**: 原子写入策略（写 tmp → rename）没有在 `Close()` 前调用 `f.Sync()`。断电时 OS 缓存未刷盘，rename 成功但 tmp 内容为空或不完整。
- **影响**: `version.json` 损坏后所有更新操作失败。
- **修复**: 在 `f.Write(data)` 之后、`f.Close()` 之前加 `f.Sync()`。

### 3. publish_tool_avalonia: `AddProjectDialogViewModel.SelectFolder()` 空引用崩溃

- **文件**: `src/PublishTool/ViewModels/Pages/AddProjectDialogViewModel.cs:65-73`
- **问题**: `folders.FirstOrDefault()?.Path.LocalPath` 可能为 null，但 `Directory.GetFiles(path, "*.exe")` 直接使用了 `path`，未做 null 检查。
- **修复**: 加 `if (path == null) return;`。

### 4. publish_tool_avalonia: `Build` 命令缺少 `PublishVersion` API 调用

- **文件**: `src/PublishTool/ViewModels/ProjectPageViewModel.cs:565-615`
- **问题**: `Build` 只创建本地 ZIP，从未调用服务端 `/api/project/publish_version`。`NewVersion` 和 `ChangeLogText` 收集了但从未发送。这是发布工具的核心功能缺失。
- **修复**: 在 `ProjectService` 中添加 `PublishVersionAsync` 方法，并在 `Build` 流程中调用。

### 5. publish-cli: `resolveConfig()` 静默丢弃配置加载错误

- **文件**: `internal/cmd/root.go:58-62`
- **问题**: `LoadProject` 和 `LoadGlobal` 的错误被 `_` 丢弃。损坏的配置文件被当作空默认值处理，用户无法得知。
- **修复**: 传播错误，至少在 debug 模式下输出警告。

### 6. publish-cli: `config init` 标记了不存在的 flag 为 required

- **文件**: `internal/cmd/config.go:22-23`
- **问题**: `MarkFlagRequired("project")` 和 `MarkFlagRequired("path")` 但 `cmdConfigInit` 只定义了 `--id` flag。`--project` 和 `--path` 是 `RootCmd` 的 `PersistentFlags`，可能导致 `config init` 命令行为异常。
- **修复**: 将 `MarkFlagRequired` 移到 `RootCmd` 上，或在 `cmdConfigInit` 上也定义这些 flag。

## 二、高 (High)

### 7. client: `check_update` 待更新状态下 `current_version == new_version`

- **文件**: `client/cmd/check_update.go:43-50`

### 8. client: `download_update` 缺少 URL/ProjectName 验证

- **文件**: `client/cmd/download_update.go:25-32`
- **问题**: 不像 `check_update` 和 `check_diff` 那样验证配置，URL 为空时产生令人困惑的网络错误。

### 9. client: 8 处 `WriteVersion` 错误被忽略

- **文件**: `client/cmd/apply_update.go:66,106,133,144`, `client/cmd/rollback.go:75,108,125,136`

### 11. server: `UpdateProject` 验证错误消息不正确

### 12. publish_tool_avalonia: `CancellationTokenSource` 泄漏

## 三、中 (Medium)

### 17. server: `DeleteProject` 删除不存在的项目静默成功

- **文件**: `server/internal/service/project_service.go:152-161`
- **问题**: `Update().Where(project.IDEQ(projectId)).SetIsDeleted(true).Save(ctx)` 在项目不存在时返回 0 affected rows 但无错误，响应仍为 `IsSuccess: true`。
- **修复**: 先查询是否存在，或检查 `Save` 返回的 affected count。

### 18. server: `GetAllFilesByProjectId` 泄露服务端绝对路径

- **文件**: `server/controllers/file_download_controller.go:77`
- **问题**: `FileAbsolutePath: absPath` 将服务端完整路径（如 `C:\server\data\projectName\file.txt`）暴露给客户端。
- **修复**: API 只返回 `FileRelativePath`，`DownloadFile` 接受 `projectId` + `relativePath`。

### 19. server: 文件上传无并发保护（竞态覆盖）

- **文件**: `server/controllers/file_upload_controller.go`
- **问题**: 两个并发上传同一 `relativeFileName` 可能互相覆盖。无文件锁机制。
- **修复**: 使用临时文件名上传 + 原子 rename。

### 20. server: 无请求大小限制

- **文件**: `server/controllers/file_upload_controller.go`
- **问题**: 无 `MaxMultipartMemory` 配置。恶意客户端可上传超大文件耗尽磁盘。
- **修复**: 设置 `router.MaxMultipartMemory` 或在 handler 中检查文件大小。

### 21. server: 服务层返回 `CommonResponse`（传输层 DTO），违反分层

- **文件**: `server/internal/service/project_service.go` 全部函数
- **问题**: Service 层直接返回 `models.CommonResponse`，混入了传输层关注点。Service 应返回 `(domain, error)`，Controller 负责包装。
- **影响**: Service 无法脱离 HTTP 上下文复用和测试。

### 22. client: `Rollback` 崩溃恢复逻辑脆弱

- **文件**: `client/cmd/rollback.go:60-86`
- **问题**: `applying` 状态下若 `mainFolder` 存在，直接 fall through 到正常回滚流程。如果上次中断遗留了部分目录，`prevVersionDir` 可能已存在，依赖 `.old` 清理模式的正确性。

### 23. client: `check_diff` 静默忽略 `LocalFileMD5Map` 错误

- **文件**: `client/cmd/check_diff.go:70`
- **问题**: `localMD5Map, _ := util.LocalFileMD5Map(mainFolder)`。`mainFolder` 不存在或不可访问时，所有服务端文件都被报告为"新增"。

### 24. publish-cli: HTTP 客户端无超时

- **问题**: `http.Client{}` 无超时设置。服务端无响应时请求永久挂起，`watch` 命令尤受影响。

### 25. publish-cli: `UploadFile` 整个文件缓冲到内存

- **文件**: `internal/api/client.go:167-215`

- **问题**: 已暂存且本地有修改的文件只出现在 unstaged 区，用户无法看到哪些已暂存。应至少将已暂存文件从 unstaged 移到 staged。

### 27. publish_tool_avalonia: `PullUpdate` 无并发保护

## 五、架构问题

### A. 服务端

1. **Service 层返回 `CommonResponse`** — 应返回 `(domain, error)`，controller 负责包装响应
2. **全局可变 `db.Client`** — 无法依赖注入，难以单元测试
3. **无中间件** — 无 CORS、无认证、无请求日志（`gin.Default()` 只提供基础 recovery）
4. **API 设计泄露服务器路径** — `GetAllFilesByProjectId` 返回 `FileAbsolutePath`，`DownloadFile` 接受绝对路径。应改为 `projectId` + `relativePath`

### B. 客户端

5. **路径分隔符处理不一致** — `filepath.Rel`（反斜杠）vs `normalizePath`（正斜杠）是 #1 号 bug 的根因
6. **版本状态机不完整** — `downloaded`/`applying` 状态下版本号语义混乱（#7），`version_previouse` 拼写错误
7. **错误恢复路径写入忽略错误** — 可导致不可恢复状态（#9）
8. **文件哈希重复计算** — MD5 和 SHA256 各读一遍文件（#33）

### C. publish-cli

9. **暂存区状态管理不完整** — `Add` 不区分 new/modified（#15），`publish` 不清空暂存区（#16），status 的 staged/unstaged 分离有误（#26）
10. **无 HTTP 超时** — `watch` 命令可能永久阻塞（#24）
11. **全局可变 flag 变量** — 多命令共享同一包级变量，测试困难
12. **配置错误静默丢弃** — `resolveConfig` 始终返回 nil error（#5）

### D. publish_tool_avalonia

13. **线程安全** — `SaveAsync` 后续在非 UI 线程触发集合修改（#14）
14. **同步 I/O 阻塞 UI** — MD5 计算、目录扫描都在 UI 线程（#29）
15. **Build 缺少 PublishVersion 调用** — 发布工具的核心功能不完整（#4）
16. **CancellationTokenSource 泄漏** — 多处创建但未 dispose 前实例（#12）

### E. 跨项目一致性

17. **JSON 字段映射** — 所有 DTO 映射已验证正确（snake_case vs camelCase 区分正确）
18. **API 路径一致** — 三端使用的 API 路径一致
19. **错误响应格式** — server 的 `CommonResponse` 与 client/publish-cli 的 `Output` 结构一致
20. **GB/GiB 单位** — server 计算为 GiB 但 Avalonia 端显示为 GB（#31）
