## 🔴 Critical（严重 — 应立即修复）

### 安全漏洞

**1. server: 路径穿越 — 任意文件读取**

`controllers/file_download_controller.go` — `DownloadFile` 直接取 `ctx.Query("path")` 传给 `ctx.File()`，无任何校验。攻击者构造 `?path=../../etc/passwd` 即可读取服务器任意文件。

**2. server: 路径穿越 — 任意文件写入**

`controllers/file_upload_controller.go` — 拼接路径时只替换了 `\` → `/`，未过滤 `../`。攻击者设置 `relativeFileName=../../etc/cron.d/backdoor` 即可写入任意路径。

**3. server: GetProjectOSInfo 数组越界 Panic**

`controllers/project_controller.go` — `cpu.Info()` 的错误被忽略，返回空切片时 `infos[0].ModelName` 直接 panic（容器/虚拟机环境常见）。

**4. server: 创建项目 TOCTOU 竞态条件**

`controllers/project_controller.go` — 名称唯一性检查（查询）和创建（写入）之间没有原子性保证，并发请求可同时创建同名项目。且数据库 `name` 字段无 UNIQUE 约束。

**5. server: 全部 API 返回 HTTP 200**

所有 controller 的错误响应都返回 `HTTP 200` + `CommonResponse{IsSuccess:false}`，包括 404、500 等。这违反了 HTTP 语义，监控/网关/中间件完全无法区分正常和异常请求。

回复：无需处理

**6. client: skipVersionUpdate 逻辑错误**

`cmd/download_update.go` — `skipVersionUpdate` 只看 `VersionStatus == "downloaded"`，不看版本号。如果用户下载了 V1.0.0，然后服务端发布 V1.0.1，再次下载时 `version.json` 不会更新，后续 apply 会应用到错误版本。

**7. client: 更新流程不处理文件删除**

`cmd/download_update.go` + `cmd/apply_update.go` — `CopyDirWithExclude` 只复制文件，不删除旧版本残留。如果服务端删除了一个有漏洞的 DLL，客户端更新后该文件仍存在，安全修补完全无效。

回复：无需处理，如果client想删除文件，应该使用脚本处理

**8. client: 断电恢复窗口数据丢失风险**

`cmd/apply_update.go` — 操作顺序是：先删旧备份 → 再 rename 当前版本 → 再 rename 新版本。如果第一步后断电，旧备份已丢失，后续步骤失败时无法回滚。

**9. publish-cli: api.Client 无超时**

`internal/api/client.go` — `&http.Client{}` 零值无超时，网络故障时请求永久挂起，CLI 工具无响应。

回复：http请求超时，每次下载文件块大小，应该都可以在配置文件中配置，也可以命令行传入

**10. publish-cli: 响应体读取错误被忽略**

`internal/api/client.go` — `io.ReadAll(resp.Body)` 的错误被 `_` 丢弃。网络中断时 body 为空，后续 JSON 解析失败但错误信息是"parse error"而非真实的网络错误，掩盖了根因。

**11. publish-cli: 不检查 HTTP 状态码**

`internal/api/client.go` — `get()`/`post()`/`UploadFile()` 均不检查状态码。服务端返回 404/500 时仍尝试解析为 `CommonResponse`，产生误导性错误。

**12. publish-cli: push 事务不完整**

`internal/cmd/push.go` — 文件上传和版本创建是两步独立操作。如果所有文件上传成功但 `PublishVersion` 失败，文件已在服务端但没有版本记录，数据不一致。

**13. publish-cli: push.go 命令间直接调用**

`internal/cmd/push.go` — `runPublish()` 直接调用 `runPush(cmd, args)`，但传入的 `cmd` 是 `cmdPublish` 而非 `cmdPush`。cobra 内部状态可能混乱。

**14. 跨项目: 三端版本号格式不统一**

- server 创建项目时自动生成 `V1.0.0`（SemVer 风格）
- client 版本比较用 `compareVersion` 按 `.` 分割数字比较
- publish-cli 的 `push` 命令版本号由用户自由输入，无格式校验
- 三端对版本号格式没有统一的约束，可能出现 `v1.0.0` vs `1.0.0` vs `20260601-1430` 的比较混乱

回复：版本号不做强约束，只比对字符串是否一致即可，不一致表示有更新

## 🟠 High（高危 — 应尽快修复）

`controllers/file_download_controller.go` — client 的 `DownloadFileWithResume` 依赖 HTTP Range 断点续传，但服务端不返回 `Content-Length`，Range 请求无法正常工作。

**17. 文件扫描时单文件失败导致整体失败**

`controllers/file_download_controller.go` — `filepath.Walk` 中一个文件读不了就 `return err`，整个文件列表接口失败。

回复：无需处理。正常是这样的呀，一个读取不了，就返回失败。

**18. 软删除项目未级联处理变更日志**

`internal/service/project_service.go` — 只标记 `project.is_deleted=true`，关联的 changelog 的 `is_deleted` 仍为 false。查询已删除项目的日志时仍然可见。

回复：无需处理。changelog与project是关联的，project被删除，其关联的changelog也查询不到

**19. 文件上传没有大小限制**

`controllers/file_upload_controller.go` — 无 `MaxBytesReader` 或 `MaxMultipartMemory` 限制，可上传超大文件耗尽磁盘。

回复：无需处理。

### client

**20. check_update 重复获取项目列表**

`cmd/check_update.go` — `FindProjectByName` 已返回含 `ForceUpdate` 字段的 project，但又调用 `GetAllProjects` 再遍历一次，完全多余的网络请求。

**21. runScript 仅支持 Windows**

`cmd/common.go` — 硬编码 `cmd /c`，Linux/macOS 上 `cmd` 不存在，`PostUpdateScript` 静默失败。

**22. ioutilReadFile 读取不完整**

`config/config.go` — 使用 `f.Read(data)` 而非 `io.ReadFull`，`Read` 不保证一次读完，大文件可能被截断。

**23. rollback 错误时状态回写错误**

`cmd/rollback.go` — 错误恢复时设置了 `VersionStatusApplied`，应该设置为 `VersionStatusDownloaded`（与 apply_update.go 保持一致）。

**24. 整个 util/process.go 仅 Windows 编译**

`util/process.go` — `// +build windows` 构建标签，Linux/macOS 上进程管理完全缺失，`apply_update` 和 `rollback` 中关闭进程的步骤会编译失败。

### publish-cli

**25. push.go 共享变量被多个命令覆盖**

`internal/cmd/push.go` — `pushVersion`、`pushMessage`、`dryRun`、`force` 四个变量被 `cmdPush`、`cmdPushAll`、`cmdPublish` 三个命令共享，flag 绑定相互覆盖。

**26. JSON 模式下人类文本混入输出流**

`internal/cmd/add.go` + `internal/cmd/project.go` — 多处 `fmt.Println` 在 `--json` 模式下直接输出人类可读文本，破坏管道 JSON 解析。

**27. CreatedAt[:10] 切片可能 panic**

`internal/cmd/project.go` — 假设 `CreatedAt` 字符串长度 ≥ 10，但格式异常或空字符串时直接 panic。

**28. 暂存文件损坏时数据丢失无警告**

`internal/staging/staging.go` — `Load()` 的 JSON 解析错误被静默忽略，返回 nil 后新文件被添加到 nil 切片上，损坏的暂存文件被覆盖写入，用户丢失所有暂存数据。

**29. config 零值歧义**

`internal/config/config.go` — `merge()` 用"非零值覆盖"策略，用户无法将 `project.id` 设为 0 或将 `server.url` 设为空字符串来覆盖全局值。应用指针类型区分"未设置"与"设置为零值"。

**30. watch 命令仅本地轮询不连服务端**

`internal/cmd/watch.go` — `watch` 仅进行本地文件轮询，不与服务端通讯，无法检测服务端文件的删除/修改。功能描述不清晰可能误导用户。

回复：暂时先去掉watch命令

**31. go.mod 版本号错误**

`go.mod` — `go 1.25.0`，Go 1.25 尚未发布，且补丁版本号写法不符合 Go 惯例。

回复：无需处理。go最新版本都1.26了

**32. 跨项目: 三端均无认证/鉴权机制**

server、client、publish-cli 全部无 token、API key 等任何鉴权，所有通信明文 HTTP。对于内网工具可能可接受，但缺少安全护栏。

回复：无需处理

**33. 跨项目: 大量 `_` 静默忽略错误**

三个项目共 15+ 处使用 `_` 丢弃关键错误（网络 I/O、文件 I/O、配置加载），导致难以排查问题。这是 Go 代码中最常见的工程质量问题。

## 🏛️ 架构层面的共性问题

### 1. 错误处理文化缺失

三个 Go 项目中共有 **15+ 处**使用 `_` 丢弃关键错误。这不是个别疏漏，而是系统性问题：
- 网络 I/O 错误被丢弃（`io.ReadAll`）
- 文件 I/O 错误被丢弃（`filepath.Walk`、`os.Stat`、`Load`）
- 配置解析错误被丢弃（`LoadGlobal`、`LoadProject`）
- 系统调用错误被丢弃（`cpu.Info`、`disk.Usage`）

**建议**: 建立 lint 规则禁止在关键路径上使用 `_` 丢弃 error，或至少记录日志。

### 2. 版本号管理缺乏统一规范

- server 创建项目时固定生成 `V1.0.0`
- client 的 `compareVersion` 按数字段比较，非数字段静默降级为 0
- publish-cli 的 push 版本号由用户自由输入，无校验
- 三端对 `V` 前缀处理不一致

**建议**: 定义统一的 SemVer 版本号规范，三端共同遵守，并在 server 端做格式校验。

### 3. 三端无任何认证/鉴权

server、client、publish-cli 全部无 token、API key、basic auth 等任何鉴权机制。所有通信明文 HTTP。对于内网工具可能可接受，但至少应预留中间件插槽。

### 4. 平台兼容性不完整

- client 的 `util/process.go` 仅 Windows 编译（`// +build windows`）
- client 的 `runScript` 仅 Windows（`cmd /c`）
- 但项目声称支持跨平台

**建议**: 要么补齐 Linux/macOS 实现，要么在文档中明确标注仅支持 Windows。

### 5. publish-cli 与 server 的"发布版本"概念不一致

- server 的 `POST /api/project/publish_version` 是一个独立端点，负责递增版本号+创建 changelog
- publish-cli 的 `push` 命令在文件上传后调用此端点
- 但 publish-cli 的 `push` 命令中，文件上传和版本创建是两步非原子操作
- 如果 publish-cli 的 `push` 中途失败，文件已在服务端但版本号未更新，数据不一致
