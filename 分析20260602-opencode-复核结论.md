## 三、架构问题（排除 publish_tool_avalonia 后）

### server

1. **Service 层返回 CommonResponse** — 应返回 (domain, error)，controller 负责包装响应
2. **全局可变 db.Client** — 无法依赖注入，难以单元测试
3. **无中间件** — 无 CORS、无认证、无请求日志（gin.Default() 只提供基础 recovery）
4. **API 设计泄露服务器路径** — GetAllFilesByProjectId 返回 FileAbsolutePath，DownloadFile 接受绝对路径。应改为 projectId + elativePath

### client

5. **路径分隔符处理不一致** — ilepath.Rel（反斜杠）vs 
ormalizePath（正斜杠）是 #1 号 bug 的根因
6. **版本状态机不完整** — downloaded/pplying 状态下版本号语义混乱（#7），ersion_previouse 拼写错误
7. **错误恢复路径写入忽略错误** — 可导致不可恢复状态（#9）
8. **文件哈希重复计算** — MD5 和 SHA256 各读一遍文件（#33）

### publish-cli

9. **暂存区状态管理不完整** — Add 不区分 new/modified（#15），publish 不清空暂存区（#16），status 的 staged/unstaged 分离有误（#26）
10. **全局可变 flag 变量** — 多命令共享同一包级变量，测试困难
11. **配置错误静默丢弃** — esolveConfig 始终返回 nil error（#5）

### 跨项目一致性

12. **JSON 字段映射** — 所有 DTO 映射已验证正确（snake_case vs camelCase 区分正确）
13. **API 路径一致** — 三端使用的 API 路径一致
14. **错误响应格式** — server 的 CommonResponse 与 client/publish-cli 的 Output 结构一致
